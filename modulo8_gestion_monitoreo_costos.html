Gestión de recursos en la nube

¿Qué es la gestión de recursos?

Administrar recursos en la nube significa controlar el ciclo de vida de instancias, almacenamiento y redes para que la infraestructura sea eficiente, confiable y no genere gastos innecesarios. No basta con lanzar recursos — hay que supervisarlos continuamente.

Señales de alerta en el uso de CPU

El uso de CPU es el indicador más inmediato del estado de una instancia. Hay dos escenarios problemáticos opuestos:

Sobrecarga
CPU > 85% sostenido
Indica que la instancia no tiene capacidad suficiente. Acción: escalar horizontalmente (más instancias) o verticalmente (instancia más grande).
Sobredimensionamiento
CPU < 10% varios días
El recurso es demasiado grande para la carga real. Genera costos sin justificación. Acción: reducir tamaño o liberar la instancia.

Escalado automático

En lugar de actuar manualmente cada vez, se configuran políticas de Auto Scaling que reaccionan a las métricas en tiempo real:

Escalado horizontal
Más instancias
Se agregan nuevas instancias cuando la carga aumenta. Distribuye el trabajo. Ideal para aplicaciones web.
Escalado vertical
Instancia más grande
Se aumenta CPU/RAM de la instancia actual. Requiere reinicio. Útil para bases de datos con carga creciente.
Regla de oro: CPU sostenida por encima del 80% durante varios minutos → activa política de escalado automático. No esperes a que el servicio se caiga.

Recursos inactivos: el costo oculto

Uno de los errores más comunes en la nube es dejar recursos corriendo sin uso. Ejemplos típicos:

RecursoProblemaAcción recomendada
Instancia EC2 detenidaSigue facturando el almacenamientoTerminar si ya no se necesita
Volumen EBS sin instanciaCosto de almacenamiento continuoCrear snapshot y eliminar
Elastic IP sin asignarCosto por IP no utilizadaLiberar la IP
Load balancer vacíoTarifa por hora aunque no tenga tráficoEliminar si no hay instancias activas

Monitoreo y métricas clave

¿Por qué monitorear?

El monitoreo convierte datos brutos de infraestructura en información accionable. Sin él, los problemas solo se detectan cuando el usuario ya los está sufriendo.

Categorías de métricas

Rendimiento
CPU, RAM, red, disco. Dicen si la infraestructura tiene la capacidad adecuada.
Disponibilidad
Uptime / downtime. Porcentaje del tiempo que el servicio está operativo.
Latencia
Tiempo de respuesta promedio. Alta latencia indica cuellos de botella.
Errores
Tasa de errores HTTP (4xx, 5xx). Indican fallos en la aplicación o infraestructura.
Tráfico
Solicitudes por segundo (RPS). Ayuda a anticipar necesidades de escalado.
Costo
Gasto por servicio, región o proyecto. Clave para optimización financiera.

Métricas de disponibilidad

La disponibilidad es la métrica más importante para acuerdos de nivel de servicio (SLA). Se calcula así:

FórmulaDisponibilidad = (Tiempo activo / Tiempo total) × 100
99.9% ("tres nueves")~8.7 h de downtime al año
99.99% ("cuatro nueves")~52 min de downtime al año
99.999% ("cinco nueves")~5 min de downtime al año
Latencia + uptime juntos son la señal más completa de salud del servicio. Un servicio puede estar "activo" pero con latencia inaceptable — ambas métricas deben estar en el dashboard.

Herramientas de monitoreo en AWS

Herramienta¿Qué monitorea?
Amazon CloudWatchMétricas de EC2, RDS, Lambda, S3. Logs, alarmas y dashboards.
AWS X-RayTrazabilidad distribuida entre microservicios. Detecta cuellos de botella.
AWS Health DashboardEstado de los servicios de AWS en tu región en tiempo real.
AWS Trusted AdvisorRecomendaciones de rendimiento, seguridad y costos.
3
Control y optimización de costos

¿Cómo se generan los sobrecostos?

Los costos en la nube crecen por tres razones principales: recursos inactivos que siguen corriendo, recursos sobredimensionados para la carga real, y falta de visibilidad sobre qué genera el gasto.

Señal de alerta: costos aumentando sin que la demanda haya crecido → lo primero es revisar reportes de uso, no aumentar el presupuesto.

Tipos de instancias para reducir costos

TipoDescripciónAhorro típicoCuándo usarlo
On-Demand Precio normal, sin compromiso — Cargas impredecibles, pruebas
Reserved Compromiso 1 o 3 años Hasta 72% Cargas estables y predecibles
Savings Plans Compromiso de gasto/hora flexible Hasta 66% Cargas estables con cierta variabilidad
Spot Capacidad sobrante de AWS Hasta 90% Cargas tolerantes a interrupciones (batch, CI/CD)

Data Lifecycle Management (DLM)

El almacenamiento es una fuente silenciosa de costos crecientes. La política de ciclo de vida automatiza el movimiento de datos hacia niveles más económicos según su antigüedad:

Datos activos
S3 Standard
→
Acceso infrecuente
S3 Infrequent Access
→
Archivo frío
S3 Glacier
→
Eliminación
Expiración
Ejemplo: logs de hace más de 90 días raramente se consultan. Moverlos a Glacier reduce el costo de almacenamiento hasta un 80% sin perder los datos.

Análisis de costo por servicio

Desagregar el gasto por servicio, equipo o proyecto permite identificar dónde hay ineficiencias y asignar presupuestos con precisión. Herramientas clave en AWS:

AWS Cost Explorer
Visualiza el gasto histórico por servicio, región o etiqueta (tag). Detecta tendencias y anomalías.
AWS Budgets
Define presupuestos por servicio o proyecto y recibe alertas cuando se supera un umbral.
Cost Allocation Tags
Etiqueta los recursos por equipo, proyecto o entorno para ver el costo de cada uno por separado.
Trusted Advisor
Detecta instancias infrautilizadas, volúmenes sin uso e IPs sin asignar que generan gasto innecesario.
4
Dashboards de monitoreo efectivos

¿Qué hace efectivo a un dashboard?

Un dashboard no es un volcado de datos — es una herramienta de decisión. Su objetivo es que cualquier persona con acceso pueda entender el estado del sistema en menos de 30 segundos.

Buenas prácticas de diseño

Hacer
Usar gráficos de líneas para tendencias en el tiempo.

Usar barras para comparar servicios o periodos.

Usar indicadores (gauges) para métricas con umbral crítico.

Agrupar métricas por categoría (rendimiento, costos, disponibilidad).

Usar colores con significado: rojo = alerta, verde = OK, amarillo = advertencia.
No hacer
Usar muchos colores sin un patrón definido.

Mostrar todos los datos sin filtros ni agrupación.

Incluir solo texto descriptivo sin visualizaciones.

Mezclar métricas de rendimiento con métricas financieras sin separación clara.

Ignorar la jerarquía visual — lo más crítico debe verse primero.

Estructura recomendada de un dashboard

SecciónMétricas incluidasTipo de gráfico
Estado generalUptime, errores activos, alertas abiertasIndicadores de semáforo
RendimientoCPU, RAM, latencia, RPSLíneas de tiempo
Disponibilidad% uptime últimos 30 días, incidentesBarra de disponibilidad
CostosGasto actual vs presupuesto, top 5 serviciosBarras + gauge de % usado
AlmacenamientoUso por bucket/volumen, datos sin accederBarras apiladas
Principio de jerarquía: las alertas críticas deben aparecer en la parte superior del dashboard. Un operador que abre el panel a las 2 AM debe ver el problema en la primera pantalla, no después de hacer scroll.
5
Alertas financieras y gestión de presupuesto

¿Para qué sirven las alertas financieras?

Las alertas automáticas convierten el monitoreo reactivo (descubrir el problema después) en monitoreo preventivo (actuar antes de que el costo se salga de control). Permiten tomar decisiones antes de que el gasto supere el presupuesto.

Interpretando el presupuesto vs gasto real

Ejemplo práctico de lectura de un dashboard financiero:

Presupuesto mensual establecido$1.200
Gasto real del mes$1.500
Diferencia (sobrecosto)+ $300 (25% sobre el límite)
Acción recomendadaRevisar recursos sobredimensionados y políticas de apagado automático
Error frecuente: aumentar el presupuesto para "cubrir" el sobrecosto sin investigar su causa. El sobrecosto es un síntoma — el problema suele ser recursos inactivos o sobredimensionados.

Configuración de alertas recomendada

UmbralTipo de alertaAcción esperada
50% del presupuesto Notificación informativa Revisar tendencia, verificar que el gasto sea esperado
80% del presupuesto Alerta de advertencia Revisar recursos activos, considerar reducción o apagado de no críticos
100% del presupuesto Alerta crítica Escalar al equipo de decisión, actuar inmediatamente sobre los mayores consumidores
CPU > 85% sostenida Alerta de rendimiento Revisar política de escalado o aumentar tamaño de instancia
CPU < 10% por 3+ días Recomendación de ahorro Considerar reducir el tamaño de la instancia (rightsizing)

Próximos pasos: hacia una nube eficiente

Hoja de ruta para una gestión madura

La madurez en la gestión cloud no se logra de un día para otro. Hay niveles progresivos que las organizaciones recorren:

NivelCaracterísticasFoco principal
Básico Monitoreo manual, sin alertas automáticas, revisión periódica de costos Visibilidad inicial
Intermedio Dashboards configurados, alertas activas, políticas de escalado definidas Reacción rápida a problemas
Avanzado FinOps implementado, rightsizing automatizado, DLM activo, presupuestos por equipo Prevención y optimización continua

Mejores prácticas consolidadas

Rendimiento
Configurar Auto Scaling con umbrales de CPU entre 60-80%. Revisar rightsizing mensualmente. Usar instancias reservadas para cargas estables.
Disponibilidad
Definir SLA mínimo (99.9% recomendado). Monitorear latencia y uptime juntos. Configurar health checks en load balancers.
Costos
Etiquetar todos los recursos por proyecto/equipo. Activar AWS Budgets con alertas en 50%, 80% y 100%. Revisar Cost Explorer semanalmente.
Almacenamiento
Implementar políticas de ciclo de vida en S3. Mover datos de más de 90 días a Glacier. Eliminar snapshots y volúmenes sin uso.
Seguridad
Usar IAM con mínimo privilegio. Activar CloudTrail para auditoría. Revisar Security Hub regularmente para detectar configuraciones riesgosas.
Automatización
Apagado automático en entornos dev/staging fuera de horario. Usar Infrastructure as Code (IaC) para reproducir entornos de forma consistente.

Resumen del módulo

Lo más importante del módulo 8: los dashboards no son un fin en sí mismos — son herramientas de decisión. El monitoreo sin alertas es ruido. Las alertas sin presupuesto son solo números. La gestión efectiva une métricas de rendimiento, disponibilidad y costo en una visión única que permite actuar antes de que los problemas impacten al usuario o al presupuesto.

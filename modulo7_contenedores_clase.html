¿Qué es un contenedor?

Definición

Un contenedor es una unidad estándar de software que empaqueta el código de una aplicación junto con todas sus dependencias (librerías, configuración, runtime) para que pueda ejecutarse de forma consistente en cualquier entorno.

Analogía: piensa en un contenedor como una caja de envío estándar — no importa si viaja en barco, tren o camión, el contenido llega igual.

Imagen vs Contenedor

Esta es la distinción más importante del módulo:

Imagen
Plantilla inmutable
Contiene el código, dependencias y configuración. No se ejecuta por sí sola. Es el molde.
Contenedor
Instancia activa
Es la imagen en ejecución. Puede haber múltiples contenedores corriendo desde una sola imagen.
nginx (imagen)
→
docker run nginx
→
contenedor activo

¿Por qué usar contenedores?

Consistencia
La app funciona igual en desarrollo, pruebas y producción.
Portabilidad
Corre en cualquier sistema con Docker instalado.
Velocidad
Arrancan en segundos, más rápido que una VM.
Aislamiento
Cada contenedor tiene su propio espacio de procesos.
2
Docker: conceptos clave

Componentes principales

ComponenteFunción
Dockerfile Archivo de instrucciones para construir una imagen. Define qué instalar, copiar y ejecutar.
Docker Engine Motor que corre en la máquina host y gestiona los contenedores.
Docker Hub Registro público de imágenes. Desde aquí se descargan imágenes como nginx, node, postgres.
Docker Network Permite que contenedores se comuniquen entre sí usando nombres de servicio en lugar de IPs.
Docker Volume Persiste datos más allá del ciclo de vida del contenedor.

Estructura de un Dockerfile

El Dockerfile es el archivo clave para construir imágenes. Al ejecutar docker build se genera la imagen.

# 1. Imagen base (punto de partida) FROM node:18 # 2. Copiar archivos del proyecto al contenedor COPY . /app # 3. Instalar dependencias RUN npm install # 4. Comando que se ejecuta al iniciar el contenedor CMD ["npm", "start"]

Comunicación entre contenedores

Docker Network permite crear redes privadas donde los contenedores se identifican por nombre:

docker network create mi_red docker run --network mi_red --name app_front frontend docker run --network mi_red --name app_back backend
Resultado: app_front puede conectarse a app_back usando su nombre, sin conocer la IP.

¿Cuándo usar VMs? Cuando necesitas aislamiento completo del SO, por ejemplo, ejecutar Windows sobre Linux. Los contenedores son ideales para microservicios y despliegues en la nube.

Kubernetes (K8s): orquestación

¿Para qué sirve Kubernetes?

Docker ejecuta contenedores individuales. Kubernetes gestiona y escala automáticamente grupos de contenedores en producción. Automatiza:

Escalado
Crea o elimina pods según la demanda de CPU o memoria (Horizontal Pod Autoscaler).
Self-healing
Si un contenedor falla, Kubernetes detecta el fallo y crea un nuevo pod automáticamente.
Actualizaciones
Despliega nuevas versiones gradualmente, sin tiempo de inactividad (rolling updates).
Balanceo de carga
Distribuye el tráfico entre los pods disponibles de forma automática.

Objetos principales de Kubernetes

Objeto¿Qué hace?
Pod Unidad mínima — uno o más contenedores que corren juntos.
Deployment Define cuántos pods deben existir y gestiona sus actualizaciones sin downtime.
ReplicaSet Garantiza que el número deseado de pods esté siempre en ejecución.
Service Expone un conjunto de pods al exterior mediante una IP o DNS estable. Sin él, los pods no son accesibles.
Namespace Agrupa recursos en un clúster para separar entornos (dev, staging, prod).
ConfigMap Almacena configuración separada del código de la aplicación.

Ejemplo: exponer pods con un Service

kind: Service spec: type: LoadBalancer selector: app: web ports: - port: 80
El Service garantiza que aunque los pods cambien de IP, siempre haya una dirección estable para conectarse.

Servicios de contenedores en AWS

Opciones principales en AWS

Servicio¿Qué es?Cuándo usarlo
Amazon ECS Elastic Container Service — orquestador propio de AWS para contenedores Docker. Equipos que ya usan AWS y quieren simplicidad sin gestionar clusters.
Amazon EKS Elastic Kubernetes Service — Kubernetes gestionado por AWS. Cuando ya usas Kubernetes o necesitas portabilidad multi-nube.
AWS Fargate Motor serverless para contenedores — no necesitas gestionar servidores (EC2). Quieres correr contenedores sin pensar en infraestructura.

Fargate vs EC2 en ECS/EKS

Fargate (serverless)
AWS gestiona los servidores. Tú defines CPU y memoria por tarea. Pagas solo por lo que usas. Ideal para cargas variables.
EC2 (tradicional)
Tú gestionas los nodos EC2 donde corren los contenedores. Más control y opciones de personalización. Mejor para cargas predecibles.

Aspectos clave en AWS

Escalabilidad
ECS y EKS integran Auto Scaling para aumentar o reducir contenedores según la demanda.
Seguridad
IAM controla permisos de los contenedores. VPC los aísla en red. ECR almacena imágenes de forma privada.
Monitoreo
CloudWatch captura métricas y logs. AWS X-Ray permite tracing distribuido entre servicios.
Registro de imágenes
Amazon ECR es el equivalente a Docker Hub en el ecosistema AWS — privado, integrado con ECS/EKS.
Flujo típico en AWS: Construyes la imagen → la subes a ECR → ECS/EKS la despliega como contenedor → Fargate gestiona los servidores → CloudWatch monitorea todo.
